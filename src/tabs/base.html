階段一：建立基地與機密 (Infrastructure & Secrets)

這是所有工作的基礎。

建立 GitHub 倉庫 (Repository)：

這就是你的「大海」。

建立 GitHub App：

這是你大腦與小龍蝦的合法身分證。

配置 Secrets：

將所有 API 金鑰存入 GitHub Secrets。

⚠️ 注意事項

Secrets 安全：

絕對不要將 API Key（如 OpenAI Key、Telegram Bot Token）直接寫在程式碼裡。一旦 Commit 上去，全球都看得到，你的錢包會瞬間變空。務必使用 GitHub Secrets。

App 權限：

在設定 GitHub App 時，遵循「最小權限原則」。例如，如果小龍蝦只需建立 Issue，就只給 Issues: Read & Write 權限，不要給 Contents: Write（寫入程式碼）權限。

外部儲存：

準備好外部資料夾（如 AWS S3, Google Drive API 帳號）。記住，GitHub Repo 不是資料庫，不要 Commit 影片或大檔案。

階段二：建立通訊與觸發 (ChatOps & Trigger)

讓你的通訊軟體能連動 GitHub。

建立 Telegram/Line Bot：

取得 Bot Token。

架設 Webhook 轉發服務 (Optional but Recommended)：

GitHub Actions 需要一個 Webhook 來觸發。雖然有 repository_dispatch，但通常需要一個中間服務（如 Cloudflare Workers、Vercel Functions）來接收通訊軟體的 Webhook，然後再呼叫 GitHub API。

實作身分驗證：

中間服務在把指令傳給 GitHub 前，必須檢查 User ID。

⚠️ 注意事項

身分驗證 (Whitelisting)：

這是最常被忽視的坑。如果你不做 User ID 檢查，任何找到你 Bot 的人都能控制你的小龍蝦去執行任務（例如，狂刷 OpenAI API 或爬取非法內容）。必須只允許你自己的 User ID。

Webhook 穩定性：

確保轉發服務有適當的超時處理和錯誤重試機制。

階段三：建立大腦 (The AI Brain)

讓 AI 分析指令並分派工作。

編寫「大腦」GitHub Action (.yml)：

這個 Action 監聽 Webhook 事件（如 repository_dispatch）。

實作 AI 邏輯：

在 Action 內使用程式碼（Python/Node.js）呼叫 AI API（Gemini/GPT）。

提示工程 (Prompt Engineering)：

給 AI 完美的指令，讓它決定下一步該做什麼工具（function calling）。

建立 Issue：

AI 決定工作後，使用 GitHub App Token 身分建立一個 GitHub Issue。

⚠️ 注意事項

提示 engineering：

這是核心。你需要告訴 AI 有哪些小龍蝦工具可用（例如：爬蟲蝦、轉檔蝦），並要求它以結構化格式（如 JSON）回覆「工具名稱」和「參數」。

Issue ID 紀錄：

建立 Issue 後，大腦 Action 需要將 Issue Number（ID）傳回給通訊軟體，讓你知道任務已受理。

階段四：建立小龍蝦工具 (The Lobster Tools)

讓具體的 Actions 去執行工作。

編寫「小龍蝦」GitHub Actions (.yml)：

每個 Action 負責一個特定工具。

監聽 Issue Events：

這些 Actions 監聽 issues: opened 事件。

條件觸發：

使用 if: github.event.issue.title == '爬蟲任務' 來決定哪隻小龍蝦動起來。

執行與回報：

小龍蝦做完事，將成果（或外部檔案連結）寫入該 Issue 的 Comment。

⚠️ 注意事項

GitHub Actions 額度：

免費版每月有執行分鐘數限制（如 2000 分鐘）。如果你的任務（如影片轉檔）很耗時，可能很快就會用完。

錯誤處理：

小龍蝦必須能捕捉錯誤（try-catch），並在 Issue 下留言失敗原因。否則，任務失敗了你也不會知道。

標籤系統 (Labels)：

小龍蝦啟動時自動幫 Issue 貼上 Processing⏳ 標籤，完成時改為 Done✅。這讓你一眼看出進度。

階段五：建立流水線與回傳 (Pipeline & Feedback Loop)

讓小龍蝦能交接工作，並將結果傳回通訊軟體。

流水線觸發 (Workflow Run)：

當小龍蝦 1 號完成時，使用 repository_dispatch 或 workflow_call 觸發小龍蝦 2 號。

狀態同步 (Optional)：

當 Issue 狀態改變時（例如被 Label 為 Done），觸發另一個 Action。

成果回傳：

小龍蝦 2 號（或狀態同步 Action）使用通訊軟體的 API，將 Issue 內的最終成果連結發送給你。

⚠️ 注意事項

上下文記憶 (State Management)：

AI 大腦沒有記憶。如果你說「處理剛剛的影片」，大腦不知道是指哪一個。你需要一個機制（例如讀取最近 5 個 Issues 的 Comment 內容），在每次發指令給大腦時一起餵給它。

網路問題：

小龍蝦回報成果時，可能遇到 Telegram/Line API 暫時掛掉。需要有適當的重試機制。
